# Expo 聊天 App 消息存储方案（建议稿）

## 1. 目标与约束

本方案用于回答三个核心问题：
- 怎么存储？
- 存储在哪儿？
- 是否需要远端存储？

同时兼顾 Expo 项目常见诉求：
- 聊天列表和消息页打开快
- 支持弱网/离线可读（至少可看历史）
- 消息发送可靠（失败可重试）
- 可扩展到多端同步与消息漫游

---

## 2. 核心结论（先给答案）

### 2.1 怎么存储？
采用 **“本地数据库 + 本地文件 + 远端消息服务”** 的分层存储：
1. **文本/结构化消息**：本地用 SQLite（推荐）
2. **图片/语音/视频等附件**：本地文件系统存路径，数据库存元数据
3. **发送状态队列**：本地持久化（pending/sending/sent/failed）
4. **会话索引与未读计数**：本地冗余表，提升列表性能

### 2.2 存储在哪儿？
1. **端内（设备本地）**
   - SQLite：消息、会话、状态队列、同步游标
   - FileSystem：媒体文件缓存/下载文件
2. **服务端（云端）**
   - 数据库：消息主存、会话关系、已读回执
   - 对象存储：图片、语音、视频文件

### 2.3 是否需要远端存储？
**建议需要**。如果只做“单设备、本地体验 demo”，可暂时不做远端；但只要涉及以下任一场景，就必须远端：
- 多设备登录同步
- 换机后聊天记录恢复
- 离线消息补偿与可靠投递
- 群聊/客服等多人会话
- 审计、风控、合规留存

---

## 3. 推荐架构（Expo 场景）

## 3.1 数据分层

- **本地热数据层（读性能优先）**
  - 会话列表（最后一条消息、未读数、置顶）
  - 当前会话近 N 条消息
  - 发送中队列

- **本地冷数据层（容量优先）**
  - 较久历史消息（分页按需加载）
  - 媒体文件（可配置淘汰）

- **远端权威层（最终一致）**
  - 消息事实源（source of truth）
  - 设备间同步与补偿

## 3.2 本地库建议

### 方案优先级
1. **首选：SQLite（Expo 生态成熟）**
   - 优点：结构化查询强、可分页、可索引、适合消息量增长
2. 备选：AsyncStorage（仅小规模/原型）
   - 不适合大量消息检索与排序

> 结论：聊天消息不建议长期仅用 AsyncStorage。

## 3.3 消息模型（建议字段）

- `id`：服务端消息 ID（最终 ID）
- `client_id`：客户端临时 ID（发送前生成，幂等用）
- `conversation_id`
- `sender_id`
- `type`：text/image/audio/video/system
- `content`：文本或结构化 JSON
- `status`：pending/sending/sent/delivered/read/failed
- `created_at_client`、`created_at_server`
- `seq`：会话内递增序号（服务端生成，用于排序与补偿）
- `is_deleted`：软删除标记

附件额外元数据：
- `local_path`、`remote_url`、`size`、`duration`、`mime`

---

## 4. 消息流转设计（关键）

## 4.1 发送流程（本地优先）

1. 用户点击发送 -> 先写本地消息（`pending`）并立即渲染
2. 入本地发送队列 -> 后台异步发服务端
3. 服务端确认后回写：
   - `pending/sending -> sent`
   - 绑定服务端 `id/seq/created_at_server`
4. 失败则 `failed`，支持手动/自动重试

收益：用户“秒发感知”，弱网也不丢输入。

## 4.2 接收流程（实时 + 补偿）

1. 在线时通过 WebSocket/推送接收新消息
2. 写本地库并更新会话列表
3. App 冷启动/断网恢复时按 `last_seq` 向后补拉

收益：可抵抗断线、进程被杀、推送延迟。

## 4.3 排序与去重

- 排序：优先 `seq`，其次 `created_at_server`
- 去重：以 `client_id` + `conversation_id`（发送幂等）或服务端 `id` 唯一索引

---

## 5. 是否“必须远端”的判断标准

## 5.1 可不做远端（仅限）

- 单机演示、临时活动页、无账号体系
- 允许卸载即丢失，且不要求消息可靠性

## 5.2 必须做远端（正式产品）

- 账号登录体系
- 需要聊天记录漫游
- 需要群聊或客服
- 需要合规与追溯

> 对“聊天 App”这一产品定位，通常默认属于“必须远端”。

---

## 6. 数据安全与合规建议

1. 传输加密：全链路 HTTPS/WSS
2. 敏感字段：按需端侧加密（密钥放安全存储）
3. 本地数据保护：
   - 退出登录时清理本地消息与媒体缓存
   - 媒体目录做容量上限与过期淘汰
4. 合规：明确消息留存周期、删除策略、用户导出能力

---

## 7. 性能与容量策略

1. 会话页分页加载（如每次 20~50 条）
2. 建索引：
   - `conversation_id + seq`
   - `conversation_id + created_at_server`
   - `status`（发送队列）
3. 媒体缓存策略：
   - 最近访问优先保留（LRU）
   - Wi-Fi 才自动下载大图/视频
4. 冷启动优化：先读本地会话摘要，再异步增量同步

---

## 8. 分阶段落地建议

## Phase 1（MVP，1~2 周）
- 本地 SQLite 消息表 + 会话表
- 发送队列与失败重试
- 远端基础 API（发消息、拉历史）

## Phase 2（可用产品）
- WebSocket 实时收发
- 增量同步（按 seq）
- 附件上传/下载与缩略图

## Phase 3（完善）
- 多端已读同步
- 撤回/编辑/引用
- 端侧加密与更细致的留存策略

---

## 9. 最终推荐（一句话）

对于 Expo 聊天 App，建议采用：
**本地 SQLite（消息与索引）+ 本地文件系统（媒体）+ 远端消息服务（权威存储与同步）**，走“本地优先、远端最终一致”的架构。

该方案在体验、可靠性、扩展性之间最均衡，适合作为正式产品路线。
